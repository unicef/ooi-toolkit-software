Trunk Based Development

Back in 2010 a branching model named Gitflow was introduced by Vincent Driessen, where the team used 2 different branches master/main and develop with several release branches and tags. It was presented as a successful branching strategy. 12 years later and hundreds of teams attempting to follow it is now considered too complex to follow.

In trunk-based development the developer takes a small batch of work, creates a branch from main and merges the changes as early as possible.

Note

Long-lived branches are the source of integration problems and complexity

The scope of work is the main difference between using long-lived feature branches and trunk-based development.

There is an implicit prerequisite that the main branch must always be in a deployable state.
